Katib (كاتب) is the writer — a bilingual, print-grade document generator that lives inside Claude Code. This walkthrough takes you from install to self-improvement in six modules.

What you'll learn
  • Install Katib and render your first PDF in under five minutes.
  • Pick the right domain, doc type, cover, and layout for any deliverable.
  • Apply your personal brand so every document looks like yours.
  • Embed print-grade diagrams, screenshots, and inline charts.
  • Let Katib learn from your corrections via reflect.py.

Before you begin

  • macOS or Linux with Python 3.10+ and uv installed.
  • Claude Code installed, with this skill registered at ~/.claude/skills/katib/.
  • An Obsidian vault (or any folder) set as your output root.
  • Optional: GEMINI_API_KEY for image-based covers (~$0.12 per cover).
Module 1 · Foundations

Install & first render

Five minutes from zero to a branded PDF. You'll install the skill, configure output paths, and render a one-pager so you have proof the pipeline works end-to-end.

Module objectives
  • Install Katib via the shipped installer or from npm.
  • Create a user config at ~/.config/katib/config.yaml.
  • Render the one-pager sample and inspect the output folder.

Steps

  1. Install the skill

    Katib ships two ways. Clone the repo and run the bundled installer, or install via npm if you prefer a one-liner.

    # Option A — installer
git clone https://github.com/jneaimi/katib.git
cd katib && bash install.sh

# Option B — npm
npm i -g @jasemal/katib && katib install

  2. Configure output & identity

    Copy the example config and tell Katib where to put generated documents. The memory.location key is where reflect.py reads its logs from.

    # ~/.config/katib/config.yaml
output:
  destination: vault
  vault_path: ~/vault/content/katib

memory:
  location: ~/.local/share/katib/memory

identity:
  author_name: "Jasem Al Neaimi"
  company: "TridenITS"

  3. Render the hello-world

    Render a business-proposal one-pager. On first run uv resolves Katib's PEP-723 deps automatically.

    uv run scripts/build.py one-pager \
  --lang en --title "Katib works" \
  --purpose "First render"

    You should see ✓ business-proposal/one-pager.en · 1 page. The output folder contains the PDF, the rendered source HTML, a manifest.md, a token snapshot, and an Obsidian-compatible .katib/run.json.

Tip

Set KATIB_OUTPUT_ROOT=/tmp/scratch before a render to send output to a scratch path — handy when iterating on templates.

Module 3 · Identity

Brand profiles — make it look like you

Every Katib render respects a four-layer cascade: domain tokens → brand profile → CLI flags → inline overrides. A brand profile is a single YAML file that swaps colours, fonts, author name, and logo without touching any template.

Module objectives
  • Write a brand profile YAML at ~/.katib/brands/<name>.yaml.
  • Pass --brand <name> to any render.
  • Understand what falls back to the domain default when the brand omits a field.

The Jasem brand, as an example

This document itself is rendered with the jasem brand — Warm Ember amber and orange, Newsreader serif, warm off-white paper. Compare the palette below to the tutorial domain default (teal on warm off-white) you'd see without a brand.

The YAML

# ~/.katib/brands/jasem.yaml
name: Jasem Al Neaimi
name_ar: جاسم النعيمي

identity:
  author_name: Jasem Al Neaimi
  email: jasem@jneaimi.com

colors:
  accent:            "#F59E0B"   # amber
  accent_secondary:  "#FB923C"   # orange
  page_bg:           "#FAFAFA"
  text:              "#18181B"

fonts:
  en:
    primary: "Newsreader"
    fallback: "Georgia, 'Times New Roman', serif"
  ar:
    primary: "Cairo"
    fallback: "Amiri, 'Traditional Arabic', serif"

logo:
  primary: logos/jasem.svg
  max_height_mm: 14

Using it

uv run scripts/build.py proposal \
  --domain business-proposal --lang en \
  --brand jasem --with-cover \
  --title "Observability SOW"
Do Omit fields you want to inherit from the domain default — partial brands are fine.
Don't Hardcode brand colours in templates. Use var(--accent) so brand swaps work.
Module 4 · Composition

Covers, layouts, diagrams, screenshots

Katib ships three cover engines, two interior layouts, print-native inline SVG diagrams, and a full screenshot pipeline for tutorials. This module is the composition cheatsheet.

Cover styles

minimalist-typographic
Free, deterministic, CSS-only. Editorial domains, formal letters, personal docs.
neural-cartography
Gemini Nano Banana 2. ~$0.12/cover. Institutional weight — proposals, formal.
friendly-illustration
Gemini Nano Banana 2. ~$0.12/cover. Warm & approachable — tutorials (the one on this document's cover).

Screenshot pipeline

Inline diagrams

Diagrams are inline SVG embedded directly in the template. WeasyPrint renders SVG natively, so they stay crisp at any print DPI, text is selectable, and colours can pick up the active brand via {{ '{{ colors.accent }}' }} interpolation. Every diagram in this very document is authored this way.

When to use what

Inline SVG for architecture, data-flows, and charts — print-grade. /diagram skill (Excalidraw) for expressive or hand-sketched content, but those rasterise for print. PNG screenshots go through the shot → annotate → frame pipeline.

Module 5 · Self-improvement

Reflect — let Katib learn from you

Every successful render appends one line to runs.jsonl. Every correction you log writes to feedback.jsonl. Doc types that don't fit land in domain-requests.jsonl. reflect.py reads all three and surfaces actionable proposals — read-only by design.

Module objectives
  • Understand the three passive-capture logs.
  • Run reflect.py with time windows and domain filters.
  • Know when the ≥3× threshold triggers each proposal type.

The three proposal types

Each fires only when a pattern recurs ≥3 times inside the active time window (default 30 days). The chart below shows why the bar matters: below three, noise; at three, signal.

Proposal types

string-swap
Same before→after correction repeats in one domain/lang. Audit templates + references/writing.<lang>.md.
new-domain-candidate
Repeated requests routed to the same closest-match domain. Consider a new doc_type or a new domain.
unused-doc-type
A doc_type in styles.json that rendered zero times in the window. Candidate for deprecation — confirm before removing.

Typical session

uv run scripts/reflect.py                       # last 30 days
uv run scripts/reflect.py --since 7d             # narrow
uv run scripts/reflect.py --domain tutorial      # filter
uv run scripts/reflect.py --propose              # write Markdown proposal file
uv run scripts/reflect.py --json                 # for scripting
reflect.py is read-only by design. It does not edit templates, tokens, or references. You apply changes manually after reviewing the proposal — that's what keeps the skill from drifting on weak signal.
Module 6 · Appendix

Command cheatsheet

Everything Katib ships as a CLI, in one place.

Render a doc

build.py <doc-type> --domain <d> --lang <l> --brand <name>

Render + cover

Add --with-cover --cover <style>. Requires GEMINI_API_KEY.

Co-locate EN+AR

Pass the same --slug to both renders.

Verify an output

build.py --verify <folder>

Lint templates

build.py --check — fast, no render.

List brands

build.py --list-brands

Generate cover

cover.py --domain <d> --style <s> --out <path>

Screenshot

shot.py web <url> --out <path>

Annotate

annotate.py <png> --arrow x,y --label "..."

Frame

frame.py <png> --style browser-safari

Reflect

reflect.py [--since Nd] [--domain <d>] [--propose] [--json]

Tests

test-all.sh • test-tutorial.sh • test-brand.sh

Environment variables

KATIB_OUTPUT_ROOT
Override output root — useful for scratch renders.
KATIB_SLUG_OVERRIDE
Force a specific slug for the next render (or use --slug).
KATIB_IMAGE_MODEL
Override Gemini cover model (default nano-banana-2).
KATIB_CACHE_DIR
Screenshot cache directory.
KATIB_DEBUG=1
Verbose build output.
GEMINI_API_KEY
Required for image-based covers.

File layout you should know

domains/<d>/
tokens.json + styles.json + templates/*.{en,ar}.html.
styles/layouts/
Shared CSS (classic, workbook, cheatsheet).
brands/
Shipped brand profiles. User profiles live at ~/.katib/brands/.
scripts/
build.py, cover.py, shot.py, annotate.py, frame.py, reflect.py, memory.py.
references/
Design, writing, production, diagrams, screenshots guides — loaded per spec tier.

Summary

You now have the full loop: install, route to a domain, pick a doc type, apply a brand, embed diagrams and screenshots, and let reflect.py surface patterns as you accumulate usage. Katib is deliberately opinionated — that's what makes the output printable without you thinking about design decisions on every render.

What's next

  • Render a real deliverable — your first client proposal, or a tutorial you've been putting off.
  • Write a second brand profile for a client and hand them ready-to-print drafts.
  • After ten or twenty renders, run reflect.py and see which patterns surface — that's how Katib learns what you need.
